# 预训练数据集清单与格式对齐

> **目标**：按 **主流 VLA / Foundation 论文** 汇总预训练用了什么数据，以及异构数据如何对齐到同一训练接口。  
> **互补**：[具身数据集全景](./具身数据集全景.md)（按数据集分层）· [Open X 与 RLDS](./Open-X与RLDS格式.md)（存储 schema）· [训练全貌 §7 Recipe](../../VLA训练与数据全貌-深度版.md)

---

## 一句话

预训练数据 ≈ **OXE 聚合真机轨迹（主粮）+ 各院私有/UMI/仿真补量 +（可选）Web VLM / Ego 语义 prior**；对齐分两层——**存储层统一到 RLDS/LeRobot episode schema**，**语义层用归一化 / Unified Action / embodiment id / 采样权重**把异构 action 变成可混训的 batch。

## 二、对齐分两层：存储对齐 vs 语义对齐

```text
各 lab 原始：HDF5 / bag / UMI / 私有
        │
        ▼ ① 存储对齐（能共同 iterate）
   RLDS Episode/Step  或  LeRobot parquet+mp4
        │   字段：observation / action / language / is_*
        ▼ ② 语义对齐（能共同算 loss）
   同一 collator 输出的 batch：
     images（resize/mask）+ tokens + actions_norm（统一维/归一化）
```

| 层次 | 解决什么 | 典型手段 |
|------|----------|----------|
| **存储对齐** | 文件能不能用同一 loader 读 | RLDS schema；LeRobot `info.json` |
| **观测对齐** | 相机数/分辨率/是否有腕部相机 | 选主相机；缺视角 mask/零图；resize 到 VLM 输入 |
| **语言对齐** | 有无指令、模板是否贫乏 | 缺省填空串；VLM 重写多种说法；step 级 vs episode 级 |
| **动作对齐** | dim / absolute·delta / joint·EEF / Hz | 见下一节四种流派 |
| **采样对齐** | 大集淹没小集、负迁移 | ∝√N、dynamic reweight、策展子集、Data Pyramid |

## 三、四大「动作格式对齐」流派（预训练时真正关键）

> **先建立直觉**：文件已经是 RLDS 了，只说明「能读到一串 float」。  
> 动作对齐回答的是：**这串 float 各自代表什么物理量、量纲差多少、最后要塞进同一个 loss 时怎么改，才能让模型在一个 batch 里混学**。  
> 四个流派不是四种文件格式，而是 **四种改写 + 归一化 + 接模型头** 的 recipe。

**总对照（读完再看细操）**：

| 流派 | 一句操作 | 模型最终吃什么 | 代表 |
|------|----------|----------------|------|
| A | 全压成同一种 7D delta，再量化成 token | 每步 7 个离散 token | OpenVLA |
| B | 按物理含义填进 128 维固定槽位，空位填 0 | 连续 128 维 + Diffusion | RDT-1B |
| C | 不强行同维；各数据集自己 normalize，出口可换 | 各头自己的维数 | Octo |
| D | 连续向量 pad 到统一长度，按平台 normalize，一次预测一长串 | H×d_a 连续 chunk + Flow | π0 |

### 3.0 动手前：同一条「抓杯子」会记成什么

假设都是「夹爪抓起杯子」的第 t 步，原始记录可能完全不同：

| 来源 | 原始 `action` 长什么样 | 物理含义 |
|------|------------------------|----------|
| Bridge / RT-1 | `[0.012, -0.003, 0.008, 0.01, 0, 0, 1.0]`（7 个数） | **相对上一步的** 位移+转角 + 夹爪开合 |
| DROID | `[0.42, 0.11, 0.35, qx,qy,qz,qw, 0.0]` 之类 | **相对世界系的绝对** 末端位姿 + 夹爪 |
| ALOHA | `[q1..q6, gL, q7..q12, gR]`（14 个数） | **左右臂关节角目标**（不是末端增量） |
| UMI | `[Δx,Δy,Δz, ΔRx,ΔRy,ΔRz, g]` | **相对上一帧夹爪系** 的位移 |

维度碰巧都是 7，**不等于可以拼进同一个 batch 直接平均**——就像「时速 60」和「海拔 60 米」不能相加。下面四个流派就是四种处理办法。

---

### 流派 A · OpenVLA：压到 7D delta → 分位数归一化 → 256 bins

**设计目标**：所有机器人最后都变成「下一步末端怎么动一点点 + 夹爪」，再交给 LLM 当文字 token 预测。

#### 具体操作流水线（按步）

```text
Step 1  语义统一（最重要，最容易错）
        DROID 绝对位姿  →  相邻帧相减或相对当前位姿，变成 delta
        ALOHA 关节角    →  正运动学 FK 得到末端位姿，再算 delta
        Bridge 已是 delta → 基本不动（只做单位/轴序检查）

Step 2  压成固定 7 维
        [Δx, Δy, Δz, Δroll, Δpitch, Δyaw, gripper]
        多出来的维丢掉；双臂通常只取一侧或合并策略（信息损失）

Step 3  按「每个数据集」统计分位数（离线算一次）
        例如对 Bridge 的 Δx 维：取出训练集所有 Δx
        → 取 1% 分位 q_low、99% 分位 q_high
        → 映射:  a' = 2 * (a - q_low) / (q_high - q_low) - 1   ∈ [-1, 1]
        夹爪类似处理；异常大值被夹到端点（quantile 的作用）

Step 4  离散化成 256 个桶
        bin = round( (a' + 1) / 2 * 255 )   →  整数 0..255
        一步动作 → 7 个整数 → 查成 7 个 vocab token

Step 5  训练
        输入: 图像 + 「pick up the cup」
        标签: 这 7 个 action token（和文字一样用 CE loss）
```

#### 数值小例子（只看 Δx 一维）

```text
Bridge 某步 Δx = 0.02 m
该维 quantile: q_low=-0.05, q_high=0.05
→ a' = 2*(0.02-(-0.05))/(0.05-(-0.05)) - 1 = 0.4
→ bin = round((0.4+1)/2 * 255) = 179
→ 变成 token_id 对应「动作桶 179」
```

若 **DROID 没做 abs→delta**，同一维可能是 `0.42`（绝对坐标），quantile 按 Bridge 算会乱套，模型学到噪声。

#### 伪代码

```python
# 离线：每个 dataset 算一次
stats[ds][dim] = (q01, q99)

# 在线：dataloader
a = to_delta_eef_7d(raw_action, ds)          # Step 1–2
a_n = quantile_to_unit(a, stats[ds])        # Step 3 → [-1,1]^7
bins = ((a_n + 1) / 2 * 255).round().long() # Step 4
labels = action_token_ids[bins]             # 7 tokens
```

| 优点 | 缺点 |
|------|------|
| 实现清晰，和 RT-1/RT-2 一脉 | 双臂/人形被压扁 |
| 开源复现路径成熟 | abs/delta 搞错会 silently 坏掉 |

### 流派 B · RDT：128 维物理槽位（填空题，不是压扁）

**设计目标**：不强迫大家都变成 7D；做一个 **固定长度的「动作说明书表格」**，每家机器人只填自己有的格子，没有的格子写 0。

#### 具体操作流水线（按步）

```text
Step 1  准备一张「128 列表格」（全体模型共用，训练前写死）
        例如:
        [0:10)   右臂 10 个关节角槽
        [10:15)  右夹爪
        [30:33)  右末端 xyz
        [33:39)  右末端 6D 旋转
        [50:60)  左臂关节 …（与右臂对称）
        [100:103) 移动底座
        其余保留位保持 0

Step 2  为每种机器人写 mapping 函数（核心工程）
        Franka 单臂 7 关节 → 填右臂槽 [0:7)，左臂全 0
        ALOHA 14D          → 右臂 [0:6)+gripper，左臂 [50:56)+gripper
        UMI 相对 EEF       → 填右 EEF 槽 [30:39)+gripper，关节槽 0
        带底盘的 RT-1 类   → 再填 [100:103)

Step 3  数值归一化（通常仍 per-dataset 或 per-embodiment）
        关节角 → 除以关节限位 或 用该集 mean/std
        填进槽位的是「归一化后的物理量」，不是 raw 编码器读数瞎塞

Step 4  附加条件（让模型知道「现在是哪台机器、多少 Hz」）
        embodiment_id → embedding（如 Franka=3, ALOHA=7）
        control_freq  → 条件输入（20Hz 与 50Hz 同维数字含义不同）

Step 5  训练 Loss
        Diffusion 在 128 维上 denoise
        实践中常对「有效维」算 MSE，零槽可用 mask 不计入
        （否则模型会浪费容量去「预测永远为 0」）

Step 6  部署到 ALOHA
        只读出左/右臂相关槽 → 反归一化 → 发关节命令
        其他槽忽略
```

#### 填表小例子

```text
原始 ALOHA action (14D):
  右臂 q=[0.1,0.2,0.0,0.5,0.0,0.1], gR=0.8
  左臂 q=[0.0,0.1,0.0,0.4,0.0,0.0], gL=0.2

写入 unified_128（示意）:
  [0:6)   ← 右臂 6 关节（归一化后）
  [10]    ← gR
  [50:56) ← 左臂 6 关节
  [60]    ← gL
  其余 100+ 维 = 0

原始 Franka 7 关节单臂:
  [0:7) ← 七个关节；左臂与 EEF 槽若本集不用则 0
  + embodiment_id = Franka
```

对比流派 A：Franka 的第 7 关节信息 **还在**；ALOHA 双臂 **两侧都还在**。代价是向量更稀疏、实现 mapping 表更繁。

槽位全文见 [RDT 笔记 §4](../RDT-Foundation-Models.md)。

| 优点 | 缺点 |
|------|------|
| 多 embodiment / 双臂友好 | 槽位表要维护；稀疏向量 |
| 微调时只激活相关维 | mask / 采样权重配置不当仍会负迁移 |

### 流派 C · Octo：不硬统一维度；换「出口」+ 各集自己 normalize

**设计目标**：共享「看图听话」的 backbone；**动作出口可以各办各的**，像同一酒店前台，退房柜台分单人房/套房。

#### 具体操作流水线（按步）

```text
Step 1  观测侧仍要对齐（相机 resize、语言 tokenize）
        动作侧：允许 Bridge 输出 7D、某 Franka 集输出 7D 但 mean/std 不同、
        另一集输出 8D（带底座）—— 不一定压成同一向量空间

Step 2  Per-dataset 归一化（几乎必做）
        Bridge:  a_n = (a - mean_B) / std_B
        DROID:   a_n = (a - mean_D) / std_D
        注意：这里 normalize 的是「该集自己的分布」，
        不要求 abs/delta 已统一到同一种语义——但实务上仍建议语义清晰，
        否则同维不同义仍会伤模型（Octo 也不能魔法消掉物理歧义）

Step 3  预训练时的「头」
        方案一：大一统 head，输入端 pad 到 max_dim，无效维 mask
        方案二：若干 dataset-group 共用一个 head（同构臂一组）
        （开源实现以论文/代码配置为准，思想是「出口可拆」）

Step 4  新机器人 finetune（Octo 卖点）
        冻住视觉-语言 backbone
        扔掉旧 action head（或只留一小段 adapter）
        用 100～数百条新机 demo，训一个「新维数」的小 head
        → 不必把新机硬塞进旧的 7D/128D 定义里重训全网
```

#### 和 A / B 的对照例子

```text
你新买了一台 8-DoF 工业臂（7 关节 + 1 特殊夹爪）：

流派 A: 必须决定扔掉哪一维，或改 8→7 的投影规则，再全量概念对齐
流派 B: 改 mapping，找 128 表里哪几个保留槽塞第 8 维，或加槽位约定
流派 C: 冻 backbone，new_head: R^8 → 动作，少量数据 ft 即可试
```

| 优点 | 缺点 |
|------|------|
| 适配新机成本低 | pretrain 时跨机迁移不如「统一语义」那么干净 |
| 不必维护一张全球物理表 | 每个新出口仍需一点标注数据 |

### 流派 D · π0：连续 chunk + 各平台归一化（不量化成 token）

**设计目标**：精细操作怕离散桶误差；一次预测 **未来一小段轨迹**（chunk），用 Flow Matching 学连续向量，而不是 7 个分类 token。

#### 具体操作流水线（按步）

```text
Step 1  选定本平台动作表示（可各自不同，但同一平台内固定）
        例: 单臂桌面用 7D delta；双臂用 14D joint；高频 UMI 用 relative EEF

Step 2  Pad / 截断到模型统一宽度 d_a
        若模型约定 d_a=32：
          7D 数据 → [a0..a6, 0,0,…,0]  后填 0
          14D     → [a0..a13, 0,…]
        （或像 RDT 有更结构化的填法；π0 系更多是工程 pad，细节多为私有）

Step 3  Per-embodiment（或 per-dataset）统计归一化
        只用该平台自己的 mean/std（或 min/max）
        A 平台的 0.02 与 B 平台的 0.02 规范化后处于各自的标准尺度

Step 4  组成 chunk（时间对齐）
        不是预测「下一步 1 个向量」，而是:
          A_chunk = [a_t, a_{t+1}, …, a_{t+H-1}] ∈ R^{H × d_a}
        π0 常见 H≈50：相当于「先规划好半秒～几秒内的动作草图」

Step 5  Flow Matching 训练（概念）
        在噪声与真值 chunk 之间采插值点，让网络预测「速度场」
        Loss ≈ MSE(预测速度, 真速度)   —— 全程连续，无 256 bins
        推理时 ODE 积几步得到整段 chunk，执行前 k 步再重新规划
```

#### Chunk 小例子

```text
控制 50Hz，H=50 → 一次规划约 1 秒
第 t 帧观测（图+指令）→ 输出 50 个 7D 动作
部署: 先执行前 25 步，在 t+25 再根据新图像重规划（action chunking）
```

对比流派 A：A 是「现在这步夹爪该开到第几个桶」；D 是「接下来一串连续轨迹长什么样」。

| 优点 | 缺点 |
|------|------|
| 精细、高频更友好 | 配方多不公开；要自建统计与 pad 约定 |
| 无量化误差 | chunk 错了会一整段偏掉，需重规划策略 |

### 四流派对照：同一批 OXE 数据会怎么走

```text
          Bridge delta 7D + DROID abs 7D + ALOHA joint 14D
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
   流派 A OpenVLA        流派 B RDT            流派 C Octo
   abs→delta / FK        填 128 槽位            各集 mean/std
   → 全变 7D             + emb_id              → 可能多头/可换头
   → quantile→bins       → Diffusion 128D      → 新机换头 ft
        │
        └──────────► 流派 D π0（若纳入）
                     pad→d_a + 平台 norm + 切成 H×d_a chunk
                     → Flow Matching（连续）
```

**怎么选（实务）**：

| 你的情况 | 优先 |
|----------|------|
| 复现 OpenVLA、单臂桌面、要 LLM 式 token | **A** |
| 双臂、多机预训、要保留物理槽位 | **B** |
| 已有通用 checkpoint，快速挂新臂 | **C** |
| 精细插拔、高频、可接受连续头工程 | **D** |

### 附：RT-2 Co-train 的「跨模态」对齐（不是机器人动作互转）

```text
同一 LLM 词表：
  机器人样本 → (image, instruction, action_tokens)
  Web VLM   → (image, QA_text_tokens)
batch 内按比例混合 → 防止忘语义 + 借用 web 知识
```

这是 **任务格式对齐**（机器人步 vs 图文问答步），**不是**上面 A–D 的 action dim 对齐。可以和流派 A 叠用（RT-2 = A 类离散动作 + web co-train）。

## 六、实践选型（你要训自己的 pretrain 时）

| 目标 | 建议数据入口 | 建议对齐方式 |
|------|--------------|--------------|
| 复现开源 7B VLA | OXE 策展子集（对齐 OpenVLA 列表） | 流派 A |
| 双臂 / 多 DoF FM | OXE + 自采 ALOHA 级 | 流派 B（RDT） |
| 快速适配新单臂 | Octo checkpoint + 少量 demo | 流派 C |
| 精细工业操作 | 自采高频 teleop（参考 π0） | 流派 D |
| 低成本 scale | UMI / RDT2 路线 + 少量 robot-aligned | Relative EEF + Data Pyramid |

**负迁移提醒**：OXE 能「共存」不同 action dim，不代表「盲目全开」就更好；2026 共识是 **策展 mixture + Data Pyramid**，而不是把 60 个子集等权扔进 loader。
