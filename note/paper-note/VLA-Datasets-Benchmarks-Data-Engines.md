## 一句话总结

这篇综述以**数据为中心**的视角系统梳理了 VLA（Vision-Language-Action）领域，核心论点：未来 VLA 的进步将更多地取决于**数据基础设施**（数据集、基准测试、数据引擎）的协同设计，而非模型架构本身。

---

## 论文动机

尽管 VLA 模型取得了显著进展，但**数据基础设施**这一核心瓶颈长期被忽视。当前面临三大矛盾：

1. **真实数据 vs 合成数据**：真实数据高质量但昂贵不可扩展；合成数据规模化但物理真实性不足
2. **评估碎片化**：缺乏标准化协议，不同工作之间难以公平比较
3. **数据生产的规模化**：如何持续生成高质量训练数据仍是一个开放问题

---

## 核心框架：三大支柱

论文围绕三个维度组织现有工作（覆盖 2023-2025）：

```
VLA 数据基础设施
├── Datasets（数据集）     → 训练材料
├── Benchmarks（基准）     → 评估标准
└── Data Engines（数据引擎）→ 规模化数据生产
```

## 一、VLA 数据集（Datasets）

### 1.1 真实世界数据集

| 数据集                   | 机器人平台           | 关键特点                                   |
| --------------------- | --------------- | -------------------------------------- |
| **Open X-Embodiment** | 22 种机器人         | 跨平台聚合，异构动作接口，适合预训练                     |
| **RT-1**              | Everyday Robots | 车队级遥操作，Action: Delta EEF               |
| **RT-2**              | Everyday Robots | 融合网络知识做联合训练                            |
| **DROID**             | Franka Panda    | 野外采集（in-the-wild），Action: Absolute EEF |
| **BridgeData V2**     | WidowX 250      | 低成本标准化设置，Action: Delta EEF             |
| **RH20T**             | 4 种机器人          | 多模态接触数据（触觉/力），Action: EEF+DoF          |
| **Ego4D**             | 人手              | 人机交互语料库，无直接动作标签                        |

**核心矛盾**：高质量（保真度）与低成本（可扩展性）之间的根本性权衡。

### 1.2 合成数据集

| 数据集                        | 关键特点                                  |
| -------------------------- | ------------------------------------- |
| **SynGrasp-1B** (GraspVLA) | 十亿级合成抓取数据，规模化几何预训练                    |
| **RoboCasa**               | 厨房仿真套件，大规模家居操作                        |
| **RoboGen**                | LLM 生成任务，Franka+足式机器人，Action: EEF+DoF |
| **MimicGen**               | 从少量人类演示增强生成海量演示                       |

**核心矛盾**：合成数据面临**保真度不足**的问题——渲染伪影、简化物理、启发式抓取姿态无法反映真实操作的稳定策略。

### 动作空间设计的两轴

1. **控制目标**：末端执行器（EEF）空间 vs 关节（DoF）空间
2. **参数化方式**：绝对目标 vs 相对增量（delta）

## 二、VLA 基准测试（Benchmarks）

论文沿**任务复杂度**和**环境结构**两个维度分析基准测试：

```
                    多场景
                      |
    CALVIN · BEHAVIOR-1K · VLABench · COLOSSEUM
                      |
    ————————————————————————————
    Meta-World · LIBERO        |
    SimplerEnv                 |
                      |
                    桌面场景
    简单任务 ←——————→ 复杂任务
```

### 2.1 简单桌面任务

- **Meta-World**：50 个简单操作任务，依赖低维状态观测
- **LIBERO**：短时域原子技能，过程性物体/空间变化
- **LIBERO-plus / LIBERO-PRO / LIBERO-X**：增强鲁棒性和公平性
- **SimplerEnv**：优先评估可靠性而非组合推理

### 2.2 复杂长时域任务

- **CALVIN**：长序列指令执行，暴露持续语义接地和时间信用分配的瓶颈
- **GemBench**：系统评估层次化泛化（新物体位置→未见实例→组合长任务）
- **COLOSSEUM**：在 14 个轴向上引入系统性视觉和物理扰动
- **VLABench**：大规模语言条件操作，长时域推理任务
- **BEHAVIOR-1K**：1000 项日常活动的以人为中心的具身 AI 基准

### 关键发现

- 长时域任务中成功率急剧下降（如 CALVIN 中 5 条指令时降至 0.08%）
- 现有基准**缺乏对 VLA 推理能力的结构化诊断**

## 三、VLA 数据引擎（Data Engines）

### 为什么需要"数据引擎"？

传统 VLA 训练数据的采集方式：

```
人类操作员 → 遥控机器人完成任务 → 录制 (图像, 动作) 对 → 这就是"数据集"
```

**问题**：这样采集的数据是**一次性、静态的**。100 小时的演示只能覆盖有限技能；新任务意味着重新采集 100 小时。

与数据集（静态、一次性采集）不同，数据引擎是**持续运行的算法管线，像工厂流水线一样源源不断生成训练数据**。三种打法各走各路：

---

### 3.1 视频到数据引擎（Video-to-Data）

**核心思路**：互联网上有海量人类操作视频（YouTube 上切菜、组装家具、拧螺丝……）。如果能把视频"翻译"成机器人可用的动作数据，就不需要人类亲自操作机器人了。

**以 Video2Policy 为例的典型管线**：

```
输入：一段人类拧螺丝的 YouTube 视频
  ↓
步骤1: 计算机视觉提取物体 mesh 和 6D 位姿
  ↓
步骤2: GPT-4o 理解"视频在做什么"，生成可执行任务代码
  ↓
步骤3: 在仿真器中执行任务代码，自动产出 (观察, 动作) 训练对
  ↓
输出：大量机器人可用的训练数据
```

> 类比：你不是亲自开赛车学驾驶，而是看 F1 比赛录像，AI 帮你把录像还原成驾驶操作指令。

各方案的区别在于"翻译方式"：

| 引擎 | 输入 | 核心技术 | 翻译方式 |
|------|------|---------|----------|
| **H2R** | 第一人称视频 | 图像修复实现手→机器人重定向 | 直接把视频中的手"修图"成机械手 |
| **RoboWheel** | 第一人称视频 | 物理感知 SDF + 残差 RL，跨形态重定向 | 不仅换手，还用物理约束确保动作可行 |
| **Video2Policy** | 互联网视频 | GPT-4o 生成可执行任务代码，88% 成功率 | 从"看懂视频"到"写出代码" |
| **X-Humanoid** | Ego-Exo4D 视频 | 视频扩散模型"机器人化"整个人体 | 给视频里的人类"穿上"机器人外壳 |
| **GenMimic** | 视频生成输出 | 加权关键点追踪+对称正则化，零样本迁移 | 完全跳过真实人类，从 AI 生成的视频学 |
| **UniSim** | 互联网视频+机器人数据 | 条件视频扩散世界模型，闭环训练 | 最极端——让策略在"预测的梦"里训练 |

---

### 3.2 硬件辅助引擎（Hardware-Assisted）

**核心思路**：不用复杂的 3D 重建，用廉价硬件**直接捕获人类动作**，然后映射为机器人指令。本质上是**动作捕捉技术在机器人领域的平民化**——从好莱坞级 mocap 棚到花 300 块 3D 打印就能搞定。

```
传统做法：人用游戏手柄/键盘遥控机器人 → 笨重、不直观
硬件引擎：人自然地用手操作 → 硬件读取动作 → 映射为机器人指令
```

| 引擎 | 成本 | 原理 | 物理-数字映射方式 |
|------|------|------|------------------|
| **ALOHA** | ~$20k | 运动学同构，双手精细操作，80-90% 成功率 | 小杆子与机器人结构一致，推杆=动机器人 |
| **GELLO** | ~$300 | 3D 打印外骨骼+被动关节正则化，可靠性提 30% | 外骨骼套手上，直接读关节角度 |
| **UMI** | GoPro 级别 | GoPro 夹爪+SLAM，3 倍采集速度 | 相机跟踪手部运动，无需固定实验室 |
| **DexCap** | 中等 | EMF 手套+RGB-D，灵巧操作 | 电磁场手套捕捉手指级精细动作 |
| **Lucid-XR** | VR 设备 | VR 头显上运行物理仿真+扩散渲染 | 人在 VR 里操作，所见即所得 |

---

### 3.3 生成式数据引擎（Generative）

**核心思路**：用 AI 在仿真中**自动创造新任务、新场景、新轨迹**，人只提供少量"种子"演示。

**最经典的 MimicGen 管线**：

```
人提供：200 条真实演示（比如"把方块从 A 放到 B"）
  ↓
步骤1: 每条演示切成"物体中心的子任务"
       ("靠近方块A" → "抓住" → "移到B上方" → "放下")
  ↓
步骤2: 对每步做空间变换
       (方块位置从 (x1,y1) 变到 (x2,y2)、柜台高度随机化)
  ↓
步骤3: 在仿真中重新执行变换后的子任务组合
  ↓
产出：200 条种子 → 5 万条新演示
```

> 类比：你教孩子一次"把杯子放到桌上"，AI 自动生成 500 种不同高度/位置的变体。

更激进如 **RoboGen**，全程零人工：
```
LLM 想任务 → LLM 写仿真代码 → LLM 写奖励函数 → 自动 RL 训练 → 产生数据
```
LLM 就是"任务设计师"。

| 引擎                               | 方法          | 关键能力                          |
| -------------------------------- | ----------- | ----------------------------- |
| **MimicGen**                     | 物体中心子任务轨迹复用 | 200 个种子→5 万条演示                |
| **DynaMimicGen**                 | DMP 适应移动物体  | 支持动态场景                        |
| **DemoGen**                      | 全合成 3D 点云编辑 | 无需真实机器人，单演示→8 任务 74.6% 成功率    |
| **GenSim / RoboGen**             | LLM 生成任务代码  | 100+ 任务库，RoboGen 平均 77.4% 成功率 |
| **RoboTwin 2.0**                 | VLM 反馈+域随机化 | 10 万+ 专家轨迹，5 种机器人平台           |
| **ROSIE / RoboEngine / EMMA**    | 扩散模型视觉增强    | 语义修复、多视角一致性                   |
| **Genie**                        | 无监督潜在动作发现   | 从互联网视频学习交互世界模型                |
| **3D-VLA / IRASim / PointWorld** | 世界模型        | 3D 目标状态生成、轨迹到视频评估             |

---

### 3.4 核心洞察：规模化失衡

论文反复强调的核心问题：

```
传统数据集：
  人 → 机器人 → 数据  （线性、受限、不可持续）

数据引擎：
  视频/LLM/仿真 → 自动化管线 → 海量数据  （指数级、可扩展）

  但根本问题：这些自动生成的数据，物理上正确吗？能迁移到真实世界吗？
```

> **数据生成规模增长快于物理落地、验证和形态对齐——存在"规模化失衡"**

这就是为什么论文主张**统一数据引擎**：不把视频、仿真、硬件当成三条独立路线，而是把它们耦合——语义生成保证多样性，物理验证保证可靠性。

## 深入：四种主流数采方法如何变成训练数据

本节从"原始信号到训练对"的角度，拆解四种主流数据采集方式的完整管线。

### A. 遥操作（Teleoperation）

**代表**：ALOHA、GELLO、RT-1

**原始数据是什么？**

人类操作主端设备（外骨骼/同构杆/VR 手柄），机器人从端同步运动。每个时间步采集到的原始信号：

```
时间步 t:
  相机拍一张 RGB 图    → ot（observation，例如 224x224x3）
  读取主端关节角       → 映射为从端动作 at（例如 7 维 delta EEF）
  语言指令             → l（例如 "pick up the red block"）
```

**关键环节：主从映射（Kinematic Mapping）**

这是遥操作最容易被忽略的环节。不同方案处理方式不同：

| 方案 | 映射方式 | 精度来源 |
|------|---------|---------|
| **ALOHA** | 运动学同构（主端和从端结构完全一致） | 机械结构保证一一对应 |
| **GELLO** | 外骨骼直接读关节角 → 线性映射到机器人关节 | 无需 IK 求解，无累积误差 |
| **VR 手柄** | 手柄位姿 → IK 解算 → 机器人关节角 | IK 引入误差，尤其在奇异点附近 |
| **键盘/游戏手柄** | 离散按键 → 固定速度指令 | 精度最差，但成本为零 |

**最终训练数据长什么样？**

```
单条轨迹（episode）：
  τ = { (o1, a1), (o2, a2), ..., (oT, aT) }  配上指令 l

训练目标（行为克隆 BC）：
  min  Σ || πθ(ot, l) - at ||²
  即：让模型看到 ot 和指令 l 时，输出的动作 πθ(ot, l) 尽可能接近人类操作的动作 at
```

**训练过程**：

```
步骤1: 采集 N 条人类遥操作轨迹（N 通常 50-500 条）
步骤2: 每条轨迹是一个 (observation, action) 序列
步骤3: 用行为克隆（BC）训练 πθ，即监督学习：
       输入：当前图片 ot + 任务描述 l
       标签：人类在这一时刻做出的动作 at
       损失：MSE（连续动作）或交叉熵（离散动作）
步骤4: 得到可以直接在真实机器人上推理的策略
```

**本质**：这是最直接的范式——**人类"示范"正确答案，模型"模仿"**。训练数据就是 (what you see, what you do) 对。

### B. 仿真采集（Simulation）

**代表**：MimicGen、RoboGen、LIBERO、RoboCasa

**原始数据是什么？**

与遥操作不同，仿真数据**不需要人类实时控制**：

```
仿真器提供一个环境：
  - 场景状态 s（物体位置、关节角等，完全已知）
  - 渲染引擎生成观测 ot（RGB 图、深度图、分割图可选）
  - 内置 IK 求解器/运动规划器/RL 训练器来产生动作 at
```

**关键环节：谁来产生动作？**

仿真数据的核心差异在于 **"教师"是谁**——谁在产生高质量的动作序列：

| 教师类型 | 原理 | 代表 |
|---------|------|------|
| **脚本专家** | 手写规则（如抓取点采样+IK），无学习成分 | RLBench, robosuite |
| **运动规划** | RRT/PRM 等算法规划无碰撞路径 | RoboGen 的子任务求解 |
| **强化学习** | 在仿真中 RL 训练收敛后导出轨迹 | RoboGen 的 RL 子任务 |
| **人类演示增强** | 从少量人类遥操作出发，空间变换生成变体 | MimicGen |

**以 MimicGen 为例，完整数据产线**：

```
输入：人类在仿真中遥操作采集的 200 条种子轨迹

步骤1：轨迹分解
  每条轨迹按"物体交互事件"切成子段：
  轨迹 = [靠近A, 抓住A, 抬起A, 移到B上方, 放下A]

步骤2：空间泛化
  对每段"抓住A"→ "移到B上方"：
    - 随机化 A 的初始位置 (x, y, z)
    - 随机化 B 的目标位置 (x', y', z')
    - 对原始轨迹做对应的空间变换（平移+旋转）

步骤3：拼接执行
  在仿真器中按顺序执行变换后的子段，
  每一步渲染一张图 ot，同时记录执行时的动作 at

步骤4：过滤
  扔掉执行失败的轨迹（物体掉落、碰撞等）

产出：200 条种子 → 5 万条新轨迹
```

**为什么仿真能自动判断成败？**

这是仿真相对于真实世界的核心优势：

```
仿真器维护了一个精确的物理状态 s_t
  → 脚本可以判断：物体 A 是否在物体 B 上方 10cm 以内？
  → 成功 = True/False，无需人工标注
  → 这就是为什么可以大规模自动化
```

**最终训练数据长什么样？**

```
和遥操作完全一样：τ = { (o1, a1), ..., (oT, aT) } + 指令 l

区别在于：
  - ot 是渲染图（可能有 domain gap）
  - at 是脚本/规划器/RL 产生的动作（可能不是最优解，但物理上可行）
  - 多了成功/失败标签（自动标注）
```

### C. UMI 式野外采集（In-the-Wild Handheld）

**代表**：UMI (Universal Manipulation Interface)

**核心问题**：UMI 的独特之处在于——**采集时根本没有机器人**。

```
传统遥操作：人有主端 → 机器人从端执行 → 记录机器人动作
UMI：人手持夹爪（上面绑着 GoPro）直接完成操作 → 没有机器人在场
```

**原始数据是什么？**

```
时间步 t:
  GoPro 拍摄的第一人称 RGB 图   → ot
  GoPro 的 SLAM 位姿估计        → 相机在空间中的 6D 位姿
  夹爪开合状态                  → 手指状态
  语言指令                      → l
```

关键：这里**没有机器人动作**，只有相机的运动轨迹和夹爪的开合。

**关键环节：从相机运动到机器人动作**

这步是 UMI 最精妙的地方：

```
步骤1: SLAM 追踪相机位姿
  每一帧 GoPro 画面对应一个 6D 相机位姿 (x, y, z, roll, pitch, yaw)

步骤2: 相机到末端执行器（EEF）的标定
  夹爪和 GoPro 是刚性连接的，所以：
  GoPro 的位姿变化 = 夹爪的位姿变化（差一个固定偏移）
  → 得到 EEF 轨迹在空间中的运动序列

步骤3: 夹爪状态提取
  从视频中检测夹爪开合（视觉分类：开 vs 关）

步骤4: 合成训练轨迹
  将 (EEF 轨迹, 夹爪状态) 作为动作 at
  将 GoPro 画面作为观测 ot
  → 得到完整的 (ot, at) 轨迹
```

**本质差异**：

```
遥操作：     人操作 → 机器人执行 → 记录机器人动作（动作是机器人"真的做了"的）
UMI：        人操作 → 只有相机运动 → 推断出"假如是机器人在做，它应该怎么动"
```

**最终训练数据和遥操作一样**，但代价低得多：
- 不需要机器人（核心成本削减）
- 不需要固定实验室（GoPro + 夹爪随身携带）
- 可以在任何场景采集（厨房、办公室、户外）

代价是精度损失——SLAM 漂移、夹爪检测失败、缺乏力反馈。

---

### D. 自我中心视频（Egocentric Video）

**代表**：Ego4D、Video2Policy、H2R、RoboWheel

**UMI 的进一步极端化**：UMI 至少还需要特制夹爪和 GoPro，自我中心视频路线连这个都省了——**直接使用人类日常活动视频**（第一人称做饭、修车、组装家具……）。

**原始数据是什么？**

```
一段 YouTube 视频 + 文字描述：
  - 连续 RGB 帧 v = [frame1, frame2, ..., frameT]
  - 音频/字幕/标题（可选）
  - 完全没有动作标签、没有相机位姿、没有夹爪状态
```

这比 UMI 还要"贫瘠"两个数量级——UMI 至少有 SLAM 轨迹，这里连空间信息都没有。

**关键环节：从视频到动作的"翻译"**

这是四条路线中**最困难**的一条，各方案走了不同的"翻译"路径：

**路径 1：结构重建（Video2Policy）**

```
输入：一段人拧螺丝的视频

步骤1: 场景理解
  对每一帧做物体检测+分割 → 提取物体 mesh + 6D 位姿

步骤2: 语义解析
  GPT-4o 看视频帧，理解任务语义：
  "这个人先用左手按住木板，右手拿起螺丝刀，
   对准螺丝孔，顺时针旋转三圈..."

步骤3: 代码生成
  GPT-4o 将语义理解翻译为仿真器中的任务代码：
  def task():
      robot.move_to(wood_board_left)
      gripper.grasp(screwdriver)
      robot.move_to(screw_hole)
      robot.rotate(clockwise, 3_turns)

步骤4: 仿真执行，产出完整 (ot, at) 轨迹
  共 100+ 个视频 → 100+ 种任务的训练数据
```

**路径 2：视觉翻译（H2R）**

```
输入：第一人称操作视频

步骤1: 逐帧"修图"
  检测人手区域 → 用扩散模型把人手修复成机械手的外观
  同时保持背景、物体、运动轨迹不变

步骤2: 动作映射
  从修复后的视频中提取机械手的"视觉运动" → 映射为动作命令

产出："看起来像是机器人做的"视频 → 从中提取动作轨迹
```

**路径 3：物理重建（RoboWheel）**

```
输入：第一人称操作视频

步骤1: 手部姿态估计
  用预训练模型提取视频中每帧的手部 3D 关节点

步骤2: 物理优化
  将人类手部轨迹放到仿真器中
  → 用 Signed Distance Function (SDF) 做碰撞惩罚
  → 用残差 RL 微调轨迹，确保物理可行
  → 支持重定向到不同形态（6/7-DOF 臂、灵巧手、人形机器人）

产出：物理上可行、可跨形态的机器人轨迹
```

**路径 4：世界模型（UniSim）**

```
输入：大量互联网视频 + 少量机器人数据

步骤1: 训练一个视频扩散模型
  输入：当前画面 + 动作描述
  输出：预测"执行这个动作后，画面会变成什么样"

步骤2: 策略在这个"模拟器"中训练
  策略 π 输出动作 → UniSim 预测下一帧 → 策略看预测结果调整下一步
  → 闭环训练，不需要真实环境

本质区别：UniSim 学的是"世界的规律"，而不是"某个具体任务怎么做"
```

**四种自监督方案的对比**：

```
方法            输入需求          动作来源            物理保真度    可扩展性
────────────────────────────────────────────────────────────
遥操作          机器人+人         人类直接示范         ★★★★★        ★★
仿真            仿真器            脚本/RL/规划         ★★★★         ★★★★★
UMI             夹爪+GoPro        SLAM推断             ★★★          ★★★★
自我中心视频     YouTube视频       GPT/扩散/优化重建    ★★           ★★★★★
```

---

### 总结：从数据到模型的统一视角

不管走哪条路线，最终殊途同归——VLA 模型的**训练接口是统一的**：

```
训练样本 = (ot, l, at)
  ot：当前的视觉观测（RGB 图像或视频片段）
  l ：语言指令（如 "pick up the red block"）
  at：机器人动作（EEF delta / DoF absolute / ...）

训练目标 = 行为克隆
  πθ(ot, l) → at
  min Σ L(πθ(ot, l), at)
```

**不同路线的本质差异只在于 (ot, at) 是怎么来的**：
- 遥操作：人类在真实机器人上示范出来的
- 仿真：脚本/RL 在虚拟环境中产生的
- UMI：从手持相机运动推断出来的
- 自我中心视频：从人类视频"翻译"出来的

这个统一视角也解释了为什么**数据基础设施**如此关键——模型架构可以通用，但数据来源决定了模型的天花板。

## 四、四大开放挑战

### 4.1 表示对齐（Representation Alignment）
异构数据源（不同机器人、动作空间、控制频率）之间的统一表示。

### 4.2 多模态监督（Multimodal Supervision）
接触丰富的操作行为需要力/触觉信号，但这些信号的代表性严重不足。

### 4.3 推理评估（Reasoning Assessment）
现有基准缺乏对时序抽象、记忆保持和多步规划能力的结构化诊断。

### 4.4 可扩展数据生成（Scalable Data Generation）
数据生成规模增长快于物理落地、验证和形态对齐——存在"规模化失衡"。

---

## 五、未来方向

1. **从扩数据规模转向弥合保真度差距**：在仿真中重建高精度真实场景
2. **统一数据引擎**：将语义生成与物理验证控制耦合，而非将视频、仿真、硬件视为独立范式
3. **高保真评估基准**：在高质量仿真环境中构建评估协议，确保训练和测试与实际部署对齐
4. **近期路径**：集成 3D 感知管道与机器人平台构建精确场景模型
5. **远期路径**：学到的世界模型实现从有限观察中自动重建和模拟真实环境
